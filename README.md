# Maimai Recruiting Coach (Public)

脉脉招聘端的人机协同技能公开版，用于把“岗位画像 -> 筛选执行 -> 候选人评估 -> 结果同步”做成可复用、可审计、可交接的标准流程。

## 1. 这个仓库解决什么问题

在实际招聘执行里，常见痛点是：
- 筛选动作不稳定，换人后效果波动大。
- 候选人评估标准不统一，同岗不同结论。
- 评估结果留痕不完整，后续复盘困难。

本仓库提供一套可落地的技能文档与脚本框架，核心目标是提升稳定性和一致性。

## 2. 适用场景

- 需要在脉脉进行批量筛选与候选人评估。
- 需要把筛选策略和评估结果沉淀到表格系统。
- 需要“AI 执行 + 人工接管”协同，而不是纯手工或纯脚本。

## 3. 核心能力

- 模板化筛选：筛选条件可版本化、可复用。
- 六项检查：候选人评估必须逐项判断并留痕。
- 双门槛同步：先过完整性，再过业务结论白名单。
- 失败兜底：验证码/风控/关键失败自动转人工。
- 脚本治理：脚本有状态分级，避免无边界扩张。

## 4. 主流程（简版）

1. 读取或生成岗位筛选模板。  
2. AI 在脉脉页面执行筛选并确认结果刷新。  
3. 按卡片顺序逐人评估，完成六项检查与依据记录。  
4. 模板级汇总后统一同步白名单结果。  
5. 回写执行结果，进入下一模板或人工接管。  

详细流程见 [skills/maimai-recruiting-coach/SKILL.md](./skills/maimai-recruiting-coach/SKILL.md)。

## 7. 稳定性与一致性约束

- 六项检查必须逐项判断并留痕。  
- `六项检查结果` 仅代表“是否判断完整”，不代表“候选人是否优质”。  
- 仅当“完整性通过 + 业务结论在白名单”时才允许同步。  
- 术语固定：`可沟通 / 储备观察 / 不合适`。  

## 8. 安全与合规边界

## 10. 常见问题

Q1：为什么强调“六项都要判断”？  
A：目标是限制漏评，确保每位候选人的判断过程完整可复盘。

Q2：为什么要“先完整性，再业务结论”？  
A：先保证记录质量，再做业务决策，避免“结论先行”导致的信息缺口。

Q3：这个仓库能直接一键运行吗？  
A：它主要提供技能规范与脚本资产；实际运行依赖你的业务环境和配置。
