# Quick Validate 清单（不改业务数据）

更新时间：2026-03-24

## 1. 版本化模板

1. 用同一基础名连续 dry-run 两次同步，确认第二次输出 `no_change_skip`。  
2. 改一个字段（如年龄上限）再同步，确认输出 `created_new_version`。  
3. 用基础名拉取，确认命中最新 `@vNNN`。

## 2. 画像生成模板

1. 用 profile JSON 生成 runtime（不写飞书）。  
2. 检查输出是否包含：关键词、公司范围、年龄/年限边界、智能筛选、招呼语要求。  
3. 若配置 OPENAI Key，检查 `llm_used` 是否按预期。

## 3. 候选人与话术

1. `sync-feishu --dry-run` 预览字段映射。  
2. 开启 `--greeting-from-template`，确认仅 `可沟通` 生成草稿。  
3. 检查草稿是否符合 `招呼语要求` 禁词约束。

## 4. 模板级一次性同步（新增）

1. 连续评估同模板多页，确认分页阶段仅产出评估文件，不触发飞书写入。  
2. 模板全部页面完成后执行一次 `sync-feishu`，确认仅写入 `可沟通`、`储备观察`。  
3. 有附件简历候选人需确认附件字段已成功上传。  
4. 人工暂停/异常中断时，确认默认不自动同步。

## 5. 成功路径复用

1. 先生成 apply-ops，手工在脉脉验证一次跑通。  
2. 用 `record-success` 记录该模板的成功路径。  
3. 用 `pull-success` 拉取路径并回放，确认无需重复试错。

## 6. 自动复用与降级（新增）

1. 同模板同签名连续执行 3 次：第 2/3 次应为 `apply_ops_source=success_path_exact`。  
2. 更换 `page-signature`：应先出现 `no_signature_match`，并降级为 `success_path_fallback` 或 `generated`。  
3. 连续上报同路径失败 2 次：路径应被标记 `deprecated`。  
4. 观察指标输出：`path_hit_rate`、`avg_apply_retry_count`、`apply_success_rate` 必须可见。  

## 7. 联想控件强约束（新增）

1. `职位名称`、`就职公司` 必须逐项执行“输入 -> 点击候选项”。  
2. 禁止使用 Enter/Tab/失焦作为联想控件选中兜底。  
3. 每点中一项后都要回读筛选摘要，确认标签已出现再继续下一项。  

## 8. 三维评估规则复用（新增）

1. 检查模板 `筛选策略` 字段已写入三维规则、权重、阈值、硬排除。  
2. 检查模板 `筛选条件JSON` 包含 `detail_evaluation_rule`。  
3. 下一轮复用同模板时，抽查 3 位候选人，确认维度定义与阈值一致。  


## 2026-03-25 六项业务检查（校正版）

1. 基础信息是否完整（至少可用于后续联系与判断）。  
2. 硬性条件是否匹配（年限/行业/职级等岗位硬条件）。  
3. 相关经验是否对口（与目标岗位核心职责匹配）。  
4. 职业稳定性是否可接受（关键经历连续性与合理性）。  
5. 求职意愿与沟通可达性是否明确。  
6. 风险备注是否完整（疑点、冲突点、待核实点是否写清）。  

验收口径补充：  
- 六项检查结果字段统一为 `六项检查结果`，取值仅允许 `通过` / `不通过`。  
- 六项检查结果=通过：六项均已判断且每项有依据（允许出现负向判断）。  
- 六项检查结果=不通过：任一项未判断/留空/仅写未知或待补充/无依据。  
- 示例：求职意愿判定为“低意向”属于已判断，不会单独导致完整性不通过。
